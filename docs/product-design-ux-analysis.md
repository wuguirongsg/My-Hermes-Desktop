# My Hermes Desktop — 产品设计体验优化分析报告

> 分析日期：2026-05-22
> 项目路径：/Users/wuguirong/sourceCode/hermes-desktop
> 当前阶段：Phase 2 — macOS 原生感（feat-201 ~ feat-206）
> 分析范围：前端 UI/UX、信息架构、交互细节、视觉一致性、错误处理

---

## 一、总体评估

项目当前处于 Phase 2 中期，核心对话闭环（Phase 0）和斜杠命令图形化（Phase 1）已基本完成。功能层面覆盖了产品规划 v3.0 中的绝大部分 P0/P1 需求。但从**产品设计体验**角度，仍有以下**7 个方向的 23 项具体优化点**值得投入。

---

## 二、信息架构与导航（4 项）

### 2.1 左侧导航栏的「文件」图标入口辨识度不足
- **问题**：当前左侧 NavBar 的「文件树」图标与 macOS Finder 的视觉语义重叠，且文件树面板与 Snapshot/Background 面板在右侧滑出，左侧图标点击后右侧出面板，空间映射不够直觉。
- **建议**：将文件树改为**左侧常驻可折叠面板**（类似 VS Code 侧边栏），而非右侧滑出。或至少改为左侧抽屉式展开，保持空间一致性。
- **优先级**：P1

### 2.2 管理面板（Dashboard iframe）的视觉割裂感
- **问题**：Dashboard 页面是全屏 iframe 嵌入 hermes dashboard（localhost:9119）。iframe 内是独立的 Web 应用，配色、字体、圆角、按钮样式与 Hermes Desktop 的 Terminal Noir / Claude Noir 主题完全不统一，用户会感到"跳到了另一个应用"。
- **建议**：
  - 短期：iframe 加载完成后通过 `postMessage` 注入 CSS 变量覆盖，至少统一背景色、字体、主色调（feat-204 的一部分）。
  - 长期：将高频管理功能（如 API Key 配置、模型选择、Skills 管理）逐步提取为原生页面，仅保留低频复杂页面用 iframe。
- **优先级**：P1

### 2.3 页面层级感知缺失
- **问题**：当前使用 HashRouter，但除了 iframe Dashboard 外，Memory / Settings / Onboarding 页面与 Chat 页面之间的切换没有过渡动画，且标题栏不显示当前页面名称（标题栏只有红绿灯 + 窗口控制）。
- **建议**：
  - 标题栏（AppTitleBar）在离开 Chat 页面时显示页面名称（如「设置」「记忆编辑器」）。
  - 页面切换增加 150ms 的淡入淡出过渡，降低认知跳跃感。
- **优先级**：P2

### 2.4 返回路径不明确
- **问题**：在 Settings / Memory / Dashboard 页面中，没有明确的「返回对话」按钮。用户需要通过左侧导航栏的「对话」图标返回，但对新用户来说不够显性。
- **建议**：在所有非对话页面的左上角（或标题栏区域）增加一个「← 返回对话」的常驻返回按钮。
- **优先级**：P2

## 三、对话体验（5 项）

### 3.1 空会话状态的引导机器人过于静态
- **问题**：GuideBot 在空会话时展示固定文案 + 每日提示轮播，但用户首次打开时可能不知道「可以做什么」。当前每日提示只是纯文本列表，没有视觉层次。
- **建议**：
  - 将每日提示按类型分组（行动型 / 查询型），使用小图标区分。
  - 增加「快速开始」卡片式布局，点击后直接填入输入框（当前似乎已有此功能，但视觉不够突出）。
  - 当检测到用户是首次使用时，GuideBot 展示更详细的欢迎语（而非随机每日提示）。
- **优先级**：P1

### 3.2 消息气泡的 hover 操作 discoverability 低
- **问题**：最后一条 assistant 消息的「重试」「撤销」按钮仅在 hover 时显示。移动端（如果未来考虑）无法 hover，且桌面端用户也可能不知道这些操作存在。
- **建议**：
  - 将消息操作栏改为**常驻轻量图标条**（半透明，hover 时增强），而非完全隐藏。
  - 或至少增加一个「更多 ⋮」常驻按钮，hover/点击后展开重试、撤销、朗读、复制。
- **优先级**：P1

### 3.3 上下文压缩触发器的时机和文案不够紧迫感
- **问题**：Context 进度条在 >70% 时显示「压缩」按钮，但按钮样式与普通按钮相同，没有紧迫感。用户可能在 context 爆了才意识到。
- **建议**：
  - 当 Context >85% 时，在输入框上方增加一条**淡入的警告条**：「上下文即将满载，建议压缩以节省 Token」。
  - 压缩按钮在 >85% 时从普通样式变为**橙色脉冲样式**。
- **优先级**：P1

### 3.4 后台任务完成后的会话跳转过于突兀
- **问题**：当前 `bg-task-done` 事件触发后，如果 session_id 存在则直接 `setActiveSessionId(payload.session_id)` 跳转。用户可能正在看其他内容，突然被切走。
- **建议**：改为**非模态通知卡片**从顶部滑入：「后台任务『XXX』已完成，[查看结果]」，让用户自主选择是否跳转。
- **优先级**：P1

### 3.5 语音输入的视觉反馈不完整
- **问题**：feat-214 已实现语音输入，但从代码看录音状态的 UI 反馈仅在麦克风按钮上体现。用户不确定是否真的在录音，以及识别到了什么。
- **建议**：
  - 录音时输入框整体变为**脉冲边框** + 声波可视化动画。
  - 实时显示识别中的文本（如「正在听：分析今天的...」）。
  - 识别完成后给一个轻微的成功动画反馈。
- **优先级**：P2

## 四、视觉一致性与设计系统（4 项）

### 4.1 DESIGN.md 与实际代码存在偏差
- **问题**：项目根目录有 `design/DESIGN.md`，描述的是 Claude 风格设计系统（暖奶油色、Copernicus 字体、珊瑚色 CTA），但实际 UI 使用的是 `index.css` 中定义的 Terminal Noir（深色琥珀金）和 Claude Noir（深色主题）。设计系统文档与实际产品视觉语言不一致，长期会导致设计债务。
- **建议**：
  - 明确产品的主打视觉语言（从代码看是 Terminal Noir 深色终端风格，但 Settings 里又有 Claude / Apple / Warp 三种主题）。
  - 如果 DESIGN.md 已不再使用，应归档或删除；如果 Claude 主题是未来方向，需要补齐 Claude 主题下的组件样式覆盖。
- **优先级**：P2

### 4.2 主题切换的即时性不足
- **问题**：useTheme 切换主题后，CSS 变量更新是即时的，但某些组件（如终端面板 xterm.js、iframe Dashboard）不会跟随主题切换。
- **建议**：
  - 终端背景（useTerminalBg）已独立设置，这是合理的；但应确保终端的前景色（文字颜色）跟随系统主题深浅。
  - iframe Dashboard 的主题融合需要推进（postMessage 注入 CSS）。
- **优先级**：P2

### 4.3 状态颜色语义不一致
- **问题**：
  - Context 进度条使用「绿→黄→橙→红」四段配色。
  - 但 GoalBar 的状态灯、后台任务状态灯、系统托盘状态灯的绿色/橙色/红色语义没有统一映射。
- **建议**：建立全局状态色板：`--status-idle` / `--status-running` / `--status-warning` / `--status-error`，所有组件复用。
- **优先级**：P2

### 4.4 字体回退策略有漏洞
- **问题**：设计文档要求 JetBrains Mono 用于代码，StyreneB/Inter 用于正文。但 CSS 中如果用户系统没有这些字体，回退链可能中断（尤其中文字体）。
- **建议**：在 CSS 中明确增加中文字体回退：`"Inter", "PingFang SC", "Hiragino Sans GB", "Microsoft YaHei", sans-serif`。
- **优先级**：P3

## 五、错误处理与反馈（3 项）

### 5.1 错误卡片的可行动性不足
- **问题**：ChatView 中的 `parseErrorCard` 将 API Key 错误、Rate Limit、模型不存在等错误转为卡片，但卡片的「前往 Dashboard」按钮实际只是 `onGoToDashboard` 回调。如果 iframe Dashboard 未就绪，点击无反馈。
- **建议**：
  - 点击「前往 Dashboard」时，先检测 dashboard 进程是否存活，未存活则显示「正在启动管理面板...」的加载状态。
  - 错误卡片根据类型显示具体的解决步骤（如 API Key 错误时，显示「1. 打开 Dashboard → 2. 点击 API Keys → 3. 粘贴你的 Key」）。
- **优先级**：P1

### 5.2 加载状态的不一致性
- **问题**：
  - 首次启动检测 Hermes CLI 时显示「正在检测 Hermes CLI…」+ loading dots。
  - 但切换会话加载历史消息时、发送消息等待 hermes 进程启动时，没有统一的加载指示器。
- **建议**：
  - 建立 3 种加载模式：1) 全局阻塞（全屏 spinner）、2) 局部阻塞（输入区禁用 + shimmer）、3) 非阻塞（顶部细进度条）。
  - 切换会话加载历史时，在消息区显示 shimmer skeleton，而非空白。
- **优先级**：P2

### 5.3 静默失败的 Rust 调用
- **问题**：代码中有大量 `.catch(() => {})` 的 invoke 调用（如 `invoke("update_tray_status", ...).catch(() => {})`）。这些错误被吞掉，用户和管理员都无法感知后台失败。
- **建议**：至少将错误输出到 `console.error`，或在开发模式下弹出 toast。生产环境可上报到日志文件。
- **优先级**：P2

## 六、性能与稳定性（3 项）

### 6.1 内存泄漏风险：事件监听未清理
- **问题**：ChatPage.tsx 中使用了大量 `window.addEventListener` + `listen()` Tauri 事件，部分 cleanup 逻辑依赖 `return` 中的函数。但 `handleNewSession` 等回调在 useEffect 依赖数组中，如果重新绑定可能导致重复监听。
- **建议**：对所有 Tauri `listen()` 返回的 unlisten 函数做严格生命周期管理，使用 `useEffect` + ref 模式确保只绑定一次。
- **优先级**：P1（稳定性）

### 6.2 轮询策略过粗
- **问题**：
  - 后台任务面板每 5s 轮询 `hermes sessions list`。
  - GoalBar 每 10s 轮询 `/goal status`。
  - 这些轮询在应用后台运行时仍在继续（即使窗口 hidden 到托盘）。
- **建议**：
  - 当窗口隐藏到托盘时，降低轮询频率（如 30s 一次）或暂停轮询。
  - 提供用户设置项「后台刷新频率」。
- **优先级**：P2

### 6.3 历史消息解析的健壮性
- **问题**：`parseHistoryMessages` 函数处理 hermes 返回的消息历史，但依赖 `tool_call_id` 的精确匹配。如果 hermes 返回的消息格式发生变化（如字段名变更），整个会话加载会失败。
- **建议**：增加 schema 校验和降级逻辑：如果无法解析，至少展示原始文本消息，而非完全空白。
- **优先级**：P2

## 七、缺失但影响体验的功能（4 项）

### 7.1 隐式上下文 info 条（feat-211，未实现）
- **影响**：用户在发送消息前，不知道自己当前的「上下文包袱」有多重（多少条历史、记忆是否加载）。这会导致用户对模型回复质量的不确定性。
- **建议**：尽快实现 feat-211，在 GoalBar 下方增加淡色 info 条：「已加载：会话历史 12 条 · 个人记忆已加载 · 工作目录 /Users/xxx/project」。
- **优先级**：P1

### 7.2 消息 Grounding 气泡（feat-213，未实现）
- **影响**：用户无法知道某条 assistant 回复是基于什么模型、什么记忆状态、什么上下文生成的。当回复出现偏差时，用户难以归因。
- **建议**：尽快实现 feat-213，在每条 assistant 消息的 header 增加 ⓘ 按钮，Popover 展示模型名、消息序号、记忆状态。
- **优先级**：P1

### 7.3 后台任务步骤时间线（feat-212，未实现）
- **影响**：当前后台任务面板只显示「最新输出摘要」，用户看不到任务的执行进度（完成了几步、当前在进行哪一步）。长任务会让用户焦虑。
- **建议**：实现 feat-212，在任务卡片展开时显示步骤时间线（✓ 已完成 / ⏳ 进行中 / ○ 待执行）。
- **优先级**：P1

### 7.4 分支会话（feat-208，未实现）
- **影响**：用户想在一个会话中探索不同方向时，必须手动复制内容新建会话。破坏了思考的连续性。
- **建议**：实现 feat-208，支持 `/branch [name]` 从当前节点分叉，侧边栏以缩进树形展示分支关系。
- **优先级**：P2

## 九、总结

My Hermes Desktop 当前的功能覆盖面已经非常扎实，产品规划的 P0/P1 需求基本到位。下一步产品设计体验的提升，核心应聚焦在：

1. **降低用户的认知负担** — 信息条（feat-211）、grounding 气泡（feat-213）、步骤时间线（feat-212）让用户始终知道"系统在做什么、基于什么在做"。
2. **消除视觉和应用间的割裂感** — Dashboard iframe 的主题融合、页面切换的过渡、返回路径的明确性。
3. **增强操作的可发现性和反馈** — 消息操作的常驻展示、语音输入的视觉反馈、错误卡片的可行动性。
4. **夯实稳定性基础** — 事件监听的生命周期管理、Rust 调用的错误处理、轮询的后台行为优化。

这些优化不需要大量新功能开发，主要是对现有功能的体验打磨和缺失信息展示层的补齐，投入产出比很高。
